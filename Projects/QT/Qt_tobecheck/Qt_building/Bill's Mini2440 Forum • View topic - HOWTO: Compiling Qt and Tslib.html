Introduction
This howto quickly shows you what I did in order to get Qt working for me on the Mini2440. I'm very new to Qt and I don't know anything more than what I'm showing you here. I'm not a coder, I'm an "installer". I gather information around the net, and pool it in one place in a hopefully easy way to understand. If you know more about this, and would like to offer tips, or tell me what I may have wrong here. Please do so.

Base System

I'm recommending that you use OpenEmbedded and to build an Ångström base image with it. Please visit the OpenEmbedded forum on this site for information on getting OE working. You should be able to use the official OE but I would recommend Michael's OE from the Mini2440 repository. It's outlined in my OE howto on this forum. His version is maintained and for the most part completely works with the mini. The official OE, while it works, isn't geared strictly for this device and you could run in to problems baking certain recipes.


Toolchain Used

I used the toolchain from OpenEmbedded from when I built the Ångström base image, which I'll talk more about in a minute. This toolchain worked better for me than the Codesourcery toolchain that I normally use. I attempted to use their 2008q3 toolchain and it needed -mtune -march and -D__GCC_FLOAT_NOT_NEEDED added to the compile line for it to work correctly and even then, I got some strange behavior. If you insist on using it, please note those options above. If you're looking at this because you've already done it, and you're getting "Invalid Instruction", well... thats why I changed to the OE toolchain. Despite people's advice about those options above, it only seemed to half-way work for me.

You can find the toolchain for OE in the /path/to/oe-build-dir/tmp/cross/. Just copy that directory to /usr/local

Code:
$ cp -R /home/bill/mini2440-oe/build/tmp/cross/* /usr/local/


You should now have an armv4t directory in /usr/local.

You can temporarily add it to your path by typing this:

Code:
$ export PATH=/usr/local/armv4t/bin:$PATH


Any toolchain you use should be in your path in order to compile tslib correctly.

Obtaining and Compiling Tslib:

Tslib is available from a git repository, so you'll need git installed. By now, you should already have this installed because its a crucial part of working with the mini since so many projects you'll need require it to access the code.

Code:
$ mkdir tslib ; cd tslib
$ git clone http://github.com/kergoth/tslib.git

Compiling tslib should be done first before Qt because you need to link against the tslib libraries. I recommend compiling it for both shared and static libraries so that the binaries come out static, and you have both versions of the libs just in case you need them. Plus, it makes the binaries more portable. Tslib isn't that large. Unless you're working on a micro filesystem, it should be fine.

Code:
$ ./autogen.sh
$ ./configure --host=arm-angstrom-linux-gnueabi --prefix=/usr/local/tslib --enable-static --enable-shared
$ make -j4
$ make install


Assuming that went well, you should have the complete set of binaries, libraries and what not, in /usr/local/tslib which is where you'll also put this on your root filesystem for the mini.

Obtaining, configuring, and compiling Qt:

You can get the latest version of Qt from this link:

http://qt.nokia.com/downloads/embedded-linux-cpp

When building Qt, you need to modify mkspecs/qws/linux-arm-g++/qmake.conf

You'll need to change the following lines so that they correspond with the path to your toolchain. If you followed my advice above, you can simply copy what I have here. Also, I added 2 lines. QMAKE_INCDIR and QMAKE_LIBDIR. Those tell it to look in the tslib directory for the libraries for tslib. And the "+" isn't a mistake, this tells the source tree that you're appending those directories to the already defined QMAKE_INCDIR and QMAKE_LIBDIR.

Code:
QMAKE_CC  = /usr/local/armv4t/bin/arm-angstrom-linux-gnueabi-gcc
QMAKE_CXX  = /usr/local/armv4t/bin/arm-angstrom-linux-gnueabi-g++
QMAKE_LINK  = /usr/local/armv4t/bin/arm-angstrom-linux-gnueabi-g++
QMAKE_LINK_SHLIB = /usr/local/armv4t/bin/arm-angstrom-linux-gnueabi-g++

QMAKE_AR  = /usr/local/armv4t/bin/arm-angstrom-linux-gnueabi-ar cqs
QMAKE_OBJCOPY  = /usr/local/armv4t/bin/arm-angstrom-linux-gnueabi-objcopy
QMAKE_STRIP  = /usr/local/armv4t/bin/arm-angstrom-linux-gnueabi-strip
QMAKE_INCDIR  += /usr/local/tslib/include
QMAKE_LIBDIR  += /usr/local/tslib/lib



Save that and run configure.

Code:
configure -embedded arm -xplatform qws/linux-arm-g++ -prefix /usr/local/Qt -qt-mouse-tslib -little-endian


I added -static and it compiled just fine, but created 2.4GB of statically compiled libraries and binaries that still required libstdc++6. So you can skip -static unless you know what you're doing. And I don't. Remember, I'm no expert. I'm just showing you what I did and how it worked for me.

You can now run make -j4 to compile the tree. It took about 45 minutes on my dual core machine.

After it's done, run make install and it should be in /usr/local/Qt ... you MUST copy that to your root filesystem in the same location. Same with tslib.


Creating your root filesystem

Please follow my instructions in the OE forum on baking a filesystem image. I used "base-image" for mine. Steps for doing this, and for copying the uImage and filesystem to an SD card or to NAND can be found on this forum. Please get your filesystem working, booting and set up before proceeding. You'll need to install libstdc++6 with ipkg or opkg from within the booted machine. I also recommend pico as well, since you'll need to edit a file or two and I hate vi.

I used an SD card for all of this. And I recommend it over NAND. It's cheaper and more durable.

Mount your SD card's root partition somewhere (i.e /mnt) on your build machine.

Code:
$ mount /dev/sdd3 /mnt


Now, copy your Qt and tslib to your root filesystem

Code:
$ cp -R /usr/local/Qt /mnt/usr/local
$ cp -R /usr/local/tslib /mnt/usr/local

You can now umount your SD card and boot up your mini with it.

If you built Qt dynamically shared, as well as tslib, you'll need to add tslib to the ldconfig. Add these lines to /etc/ld.so.conf and run ldconfig so that the system knows where the libraries are.

Code:
/usr/local/tslib/lib
/usr/local/Qt/lib


Code:
$ ldconfig


Now create the touchscreen device as well as a mouse device in your /dev/input directory.

Code:
$ mknod c 13 65 /dev/input/touchscreen0
$ mknod c 13 32 /dev/input/mice


Now, assuming everything is compiled correctly, and in its proper place, you should be able to run some Qt demos.

Testing Qt by running the Fluid Launcher

Fluid launcher is a really cool way of showing just what Qt can do visually.

Create the following shell script and name it startfluid.sh. You may need to install bash on your system. (ipkg install bash)

Example Fluidlauncher Script:
Code:
#!/bin/bash
export TSLIB_TSEVENTTYPE=INPUT
export TSLIB_CONSOLEDEVICE=none
export TSLIB_FBDEVICE=/dev/fb0
export TSLIB_TSDEVICE=/dev/input/touchscreen0
export TSLIB_CALIBFILE=/etc/ts.calib
export TSLIB_CONFFILE=/usr/local/tslib/etc/ts.conf
export TSLIB_PLUGINDIR=/usr/local/tslib/lib

if [-f -etc-ts.calib]; then
  echo "Touchscreen already calibrated..."
else
  echo "Calibrating Touchscreen..."
  /usr/local/tslib/bin/ts_calibrate /etc/ts.calib
fi

export QWS_MOUSE_PROTO=tslib:/dev/input/touchscreen0
cd /usr/local/Qt/demos/embedded/fluidlauncher ; ./fluidlauncher -qws


Change permissions on the script filename by using chmod to add the ability to execute it.

Code:
$ chmod +x startfluid


Now go ahead. Run it. See if it works. If everything is done correctly, it should bring up the calibration for the display first, and then bring up a pretty neat looking demo.


Adjusting Fonts:

I thought the font was way too tiny for my display. You can add this to the script above after the -qws switch.

Code:
-fn helvetica


Please see this site for more information. http://doc.trolltech.com/4.6/qt-embedded-running.html#command-line-options

Final Thoughts

I've seen a lot of sites talk about how easy this is to get going. Well I didn't find that to be very true. I fought with the compiler, and missing packages the whole way. And you'll probably run in to a few things as well depending on what you already have installed, and what distro you have etc.. I used Ubunti 9.04 to compile this. It took me 2 days following someone else directions to finally get it working the way I wanted. I think Qt is probably the better solution over running X11 or Gnome or whatever on the Mini2440. I think it's faster and has a lot of stunning effects. And there are a lot more things you can do with it that are way beyond my ability or the scope of this howto, but you can view the Trolltech website to learn more about it. Please share your successes and failures with me. And as always, I hope this has helped you in some way.

Thanks,
