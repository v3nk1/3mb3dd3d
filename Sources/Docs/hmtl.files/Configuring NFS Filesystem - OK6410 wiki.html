Configuring NFS Filesystem

From OK6410 wiki

Jump to: navigation, search

Probably the first question to answer is "Why NFS?". The answer is simple for several reasons:

  • You don't need to reprogram the NAND Flash every time you changed a little peace of code
  • Flash reprogramming takes a lot of time
  • NAND Flash is not endlessly programmable
  • Because the NFS server is located on your host computer, you can configure your development softwre (Eclipse, QT Quick,...) to work directly inside the NFS directory. This way every build can immediately be tested on the hardware

Used materials

  • An OK6410 board, programmed with the U-boot image and Kernel image
  • A complete filesystem build by Buildroot (as explained here)
  • Install the necessary packages: 
$ sudo apt-get install portmap nfs-kernel-server

Create and populate a NFS folder

The preparations exists of two steps:

  • Creating and populating a NFS root directory in which the root filesystem will be shared over a standard Ethernet network.
  • Making the necessary changes to the NFS server to share the correct directorie(s)

Create the root filesystem

Because of some special file permissions it is necessary that Buildroot creates a rootfs.tar file which we will use as a base for our NFS server. So first we have to configure Buildroot to create the tarred root filesystem. Go into your buildroot directory, and type: 

$ make menuconfig

In the menu make the following changes: 

Filesystem images --->
      [*] tar the root filesystem
              Compression method (no compression) --->
            ()Other random options to pass to tar

Make the complete system including a rootfs.tar file. 

$ make

After the make is complete you will notice a rootfs.tar inside the buildroot/output/images folder. The next thing to do is to create a NFS directory which will function as root. In your home directory create a folder called nfs_root or something like that. Here we will store the complete (untarred) filesystem: 

$ mkdir ~/nfs_root

IMPORTANT: People who are using an encrypted home folder can encounter problems accessing the nfs_folder inside your home directory. A good alternative is just to make a /nfs_root folder. Remember this, because the following tutorials will use ~/nfs_root as directory!

Populate the NFS folder

Now we copy the rootfs.tar to this directory (check if the buildroot directory is ok, maybe you used another version): 

$ sudo cp ~/buildroot-2011.05/output/images/rootfs.tar ~/nfs_root/

Go into your nfs_root direcotry and untar the archive: 

$ cd ~/nfs_root
$ sudo tar -xf rootfs.tar

Now the nfs_root directory is filled with the directory structure and files created by Buildroot.

REMARK: Remember that when you want to replace the filesystem by a freshly built one, you have to remove everything with sudo rights! Remove the files, copy the tar and untar it again (all using the sudo prefix). Reboot the board and your up and running again! 

$ cd ~/nfs_root
$ sudo rm -rf *

Known Errors

ttySAC0: no such device

It could be, when using the 2.6.36 kernel version, that the OK6410 board stops after it's boot procedure by saying: 

can't open /dev/ttySAC0: No such file or directory

This can be avoided by editing the Udev rules file: 

cd ~/nfs_root/etc/init.d
sudo gedit S10udev

And add the following line after start): 

mknod -m 0666 /dev/ttySAC0 c 204 64

So that it looks like: 

case "$1" in
    start)
        mknod -m 0666 /dev/null c 1 3
	mknod -m 0666 /dev/ttySAC0 c 204 64
        echo -n "Populating $udev_root using udev: "
        echo -e '\000\000\000\000' > /proc/sys/kernel/hotplug
        $UDEV_BIN -d || (echo "FAIL" && exit 1)
	$UDEVSTART_BIN || (echo "FAIL" && exit 1)
        echo "done"
        ;;
    stop)
        # Stop execution of events
        udevcontrol stop_exec_queue
        killall udevd
        ;;
    *)
        echo "Usage: $0 {start|stop}"
        exit 1
        ;;
esac

Configure your NFS server

This step you don't have to redo when you replace the contents of the nfs_root folder with a new file system!!

Time to make the necessary changes to your host system, so the nfs server is configured proparly: 

$ sudo gedit /etc/exports

And add the follwing line at the end of the file (replace your-user-name into the right user name of course ;-) ): 

/home/your-user-name/nfs_root *(rw,sync,no_root_squash,no_subtree_check)

Start the NFS service: 

$ sudo /etc/init.d/portmap restart
$ sudo /etc/init.d/nfs-kernel-server restart

Setting up OK6410 for NFS

We will discuss 2 methods for the OK6410 board to connect to the NFS server:

  • Using automatically assigned IP adresses (DHCP) (When your OK6410 is connected to a network using a router)
  • Using fixed IP's. (When your board is directly connected to your host computer)

Using DHCP TCP/IP settings

If your OK6410 board is connected to the network using a router with automatically assigned IP adresses using a DHCP server. You have to reconfigure the Linux kernel to activate DHCP. First of all go to your buildroot directory and type: 

$ make linux26-menuconfig

And make the following changes: 

[*] Networking support --->
          Networking options --->
              [*]  IP: kernel level autoconfiguration
              [*]       IP: DHCP support
              [*]       IP: BOOTP support
              [ ]       IP: RARP support

Rebuild your buildroot system: 

$ make

And flash the new kernel to the NAND Flash using the DNW tool. See the follwing article link. Open a minciom terminal, reboot the board and abort the U-boot autoboot. Inside the U-boot console we must reconfigure the boot arguments.

First of all we need to know the IP adress of your host computer. You can discover this by opening another terminal and typing: 

$ ifconfig

Using fixed IP settings

This method is not thoroughly checked! So you might have to make some changes to the boot arguments yourself. The only thing to do to use fixed IP addresses is to change the boot arguments of U-boot. First you can also use ifconfig to find the correct IP address of your host computer: 

$ ifconfig 
eth0      Link encap:Ethernet  HWaddr 00:26:18:32:2a:83  
         inet addr:192.168.1.14  Bcast:192.168.1.255  Mask:255.255.255.0
         inet6 addr: fe80::226:18ff:fe32:2a83/64 Scope:Link
         UP BROADCAST RUNNING MULTICAST  MTU:1500  Metric:1
         RX packets:17324 errors:0 dropped:0 overruns:0 frame:0
         TX packets:22222 errors:0 dropped:0 overruns:0 carrier:0
         collisions:0 txqueuelen:1000 
         RX bytes:10175604 (10.1 MB)  TX bytes:13723245 (13.7 MB)
         Interrupt:68 

lo        Link encap:Local Loopback  
         inet addr:127.0.0.1  Mask:255.0.0.0
         inet6 addr: ::1/128 Scope:Host
         UP LOOPBACK RUNNING  MTU:16436  Metric:1
         RX packets:248 errors:0 dropped:0 overruns:0 frame:0
         TX packets:248 errors:0 dropped:0 overruns:0 carrier:0
         collisions:0 txqueuelen:0 
         RX bytes:23492 (23.4 KB)  TX bytes:23492 (23.4 KB)

Knowing the inet addr:192.168.1.14 (in my case), is the info we need to set the U-boot environment. Open a minicom, reset the board and abort the autoboot procedure. Enter the following boot arguments (of course change into the right IP's: 

SMDK6410 # setenv bootargs "root=/dev/nfs nfsroot=192.168.1.14:/home/your-user-name/nfs_root,v3 ip=192.168.1.15:192.168.1.14:192.168.1.14:255.255.255.0:arm9board.net:eth0:off console=ttySAC0,115200"

For more info about the fixed IP setup, you can read more in the included Linux user manual on the DVD!
