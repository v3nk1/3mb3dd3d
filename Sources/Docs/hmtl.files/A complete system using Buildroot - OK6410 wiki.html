A complete system using Buildroot

From OK6410 wiki

Jump to: navigation, search

Buildroot is nothing more than a collection of Makefiles. By using a menuconfig command you can choose all the elements that you want to include in your build. For this tutorial I used the buildroot-2011.05.tar.gz file. You can decide to use another version of Buildroot, but it could be possible that some options have changed in newer versions.

Downloading the kernel packages

Depending on the version of the kernel you want to use you need to select a directory on the main page of the FTP website:

ftp://wikiusers@ftp.arm9board.net/ (password: Arm9board)

You have two directories: 2.6.28 and 2.6.36, and you need to choose which version of the kernel and corresponding U-boot version you want to use and download.

Configuring buildroot

First you must install the necessary packages: 

$ sudo apt-get install libncurses5-dev bison flex texinfo

IMPORTANT: 64-bit Ubuntu Users have to install the 32-bit libraries to work with the CodeSourcery Toolchains!! This is done with the following command: 

$ sudo apt-get install ia32-libs

After downloading Buildroot, copy it to your home folder and untar it: 

$ tar -xf buildroot-2011.05.tar.gz

Go into the freshly created directory: 

$ cd buildroot-2011.05

Inside the buildroot directory, create a dl folder where we can store the specific u-boot, and kernel source packages for the OK6410 board. 

$ mkdir dl

Drag the OK6410_linux.2.6.28_v1.2.tgz and uboot1.1.6_OK6410_V1.0.tgz (for 2.6.28) or OK6410_linux2.6.36.2.tgz and u-boot1.1.6_TE6410_2636.tgz (for 2.6.36) from the FTP server into the dl folder. After this we only need to tell buildroot what the name is of the packages to look for. Now we need to make a little change inside the linux tar package. Open it using the Archive manager:

Defconfig for the 2.6.28 kernel version

Double click on the OK6410_CFG file to open it. Go back to the archive mounter and look for the following file: 

arch/arm/configs/smdk6410onenand_defconfig

Open the file and replace the complete content with the content of the OK6410_CFG file. Save the smdk6410onenand_defconfig file and close the archive manager (after the archive has updated the new file).

Defconfig for the 2.6.36 kernel version

When the archive is opened select the OK6410_CFG file and copy it. Move into the following folder: 

arch/arm/configs

And paste the file here. After pasting it, rename it to the following: 

smdk6410onenand_defconfig

Wait until the archive is updated and you're ready

Buildroots menuconfig

Now we can start changing Buildroot. In a terminal, go into the Buildroot directory and use the make menuconfig command to start adapting Buildroot to your needs: 

$ make menuconfig

And make the first basic changes: 

Target Architecture (i386) --->
              (X) arm

Target Architecture Variant (generic_arm) --->
              (X) arm1176jzf-s

Target (EABI) --->
              (X) EABI

Now make the settings for the Toolchain, go back to the top menu and change the following sections: 

Toolchain --->
         Toolchain type --->
              (X) External toolchain
         Toolchain (CodeSourcery ARM 2011.03) --->
              (X) CodeSourcery ARM 2010q1
         [*] Download toolchain automatically
         (In this menu you can select gdb options if you want to)

Time to make the system configuration. Inside the top level menu, make the following changes: 

System configuration --->
         (Choose a system hostname and banner if you want to)

         (ttySAC0) Port to run a getty (login prompt) on
         Baudrate to use (115200)

Configure software parameters

Still inside the buildroot configuration, go back to the top level and make the package selection for the target: 

Package selection for the target --->
         [*] Busybox
                Busybox version (BusyBox  1.18.x)
         (The rest of the options we'll discuss later)

If you ever want to, you can always change the specific busybox settings and build packages by typing the following command in the terminal: 

$ make busybox-menuconfig

The rest of the packages will be discussed later. First let's select what filesystems Buildroot has to create. This filesystem contains all the packages you have selected for the board. You can always read the article about Flash filesystems on the Free-electrons website. But for now we will create a cramfs image: 

Filesystem images --->
          [*] cramfs root filesystem
          [ ] tar the root filesystem (Unselect this option)

For the 2.6.28 linux version

Configure U-boot:

                 U-boot version (Custom tarball) --->
                (uboot1.1.6_OK6410_V1.0.tgz) URL of custom U-Boot tarball

Making the changes to the kernel

Now to complete, exit buildroot and save your configuration. You still need to select the right LCD-driver for your Linux Kernel. To configure the kernel type: 

$ make linux26-menuconfig

And make select the right LCD-driver, as explained here in the section Choosing your LCD type.

Build everything

To start we need to download all the necessary packages and tools you've selected. You can do this by going (if you're not there anymore) to the main buildroot directory and type: 

$ make source

This can take some time to finish and after this you can start compiling: 

$ make all

It could be that you need to install some additional packages for Buildroot to work properly. Buildroot will ask this before compiling.

Inside the Buildroot directory, the compilation process created the output directory. Let's have a quick look to the structure of this directory: 

Output
  |
  | - Build: Contains all the extracted sources from the dl folder and the build files
  | - Host: Contains the binaries and files from the selected toolchain
  | - Images: Contains the files for the board: U-boot.bin, zImage and rootfs.cramfs
  | - Staging: You can ignore this folder for now
  | - Stamps: In this case, this folder will only mention that we used an external toolchain
  | - Target: Contains the directory structure for the board. The mkcramfs tool used this folder. Good to use as NFS filesystem.
  | - Toolchain: Only used if you created a specific toolchain.

Program the board

Let's assume here that the OK6410 board has never been used. The programming will be explained from zero on. A complete tutorial on the programming of the board can be found on the following two Wiki pages:

  • Programming flash using only OpenOCD
  • Programming flash using OpenOCD and DNW

I used the first for programming U-boot and the second method for programming the kernel image and the root filesystem.

I'll give you all the commands to be short:

Power up the board and open a terminal 

$ sudo openocd -f ok6410.cfg

Open another terminal to log in on openOCD: 

> nand erase 0 (if all data needs to be erased)
> nand write 0 /home/your-user-name/buildroot-2011.05/output/images/u-boot.bin 0

open another terminal, load the USB2Serial drivers and start minicom 

$ sudo minicom

For the 2.6.28 version of the kernel 

SMDK6410 # dnw c0008000

open another terminal and send the kernel zImage to the board 

$ cd ~/dnw
$ sudo ./dnw ~/buildroot-2011.05/output/images/zImage

SMDK6410 # nand write.e c0008000 100000 300000
SMDK6410 # dnw c0008000

In the DNW terminal, type: 

$ sudo ./dnw ~/buildroot-2011.05/output/images/rootfs.cramfs

Program the filesystem in flash through the minicom terminal: 

SMDK6410 # nand write.e c0008000 400000 5000000

For the 2.6.36 version of the linux kernel

Stop the boot procedure of the board and inside the minicom terminal, press the k key, put the OTG cable into the board and connect it to the host PC. Move to a new terminal, go into the DNW directory and upload the zImage to the board: 

$ cd ~/dnw
$ sudo ./dnw /the/location/of/zImage

Boot the board

Reset the board and you'll see the buildroot login.

Buildroot login: root Password: none

The next subject will focus on the use of a NFS (based on the output/target directory) filesystem. This is for two major reasons:

  • The NAND Flash filesystem is readonly mounted
  • With a NFS filesystem you don't have to re-program the board every time you change something inside the filesystem
  • If you have an error regarding ttySAC0 nu such device or directory, the solution can be found here.
