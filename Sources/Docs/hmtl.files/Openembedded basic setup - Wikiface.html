What is it

Openembedded ("OE" for short) is a tool usable to cross-compile Linux distributions on your PC. The typical target of the system is - as the name suggests - the embedded world.

The official site is at http://www.openembedded.org/ where plenty of documentation can be found.

Why another guide?

On the net you can find plenty of guides on how to install and setup OE; some of them are outdated, some tricky, some referring to other distros, some runs only on 32 bits systems not on 64 bits.

Structure

OE is built on three pillars:

  • Bitbake;
  • The recipes;
  • local.conf configuration file.

Bitbake

Taking from the official website, BitBake is, at its simplest, a tool for executing tasks and managing metadata, so it's the responsible who makes the dirty job; this tool written in Python.

Recipes

A collection of file with a .bb extension that describes what BitBake has to do; in this case where to find what's needed to build, how to configure it, build it and install it.

local.conf

This is actually the configuration file describing the target board, the release of the distro to compiles, the directory to store the source files, to compile, to install, the root image type to generate.

Install

We start from a basic debian 6.0 (Squeeze) installation and we will use a 64 bit system. The installation procedure doesn't change a lot except for the installation of 32-bit compatibility libraries that are not needed on a native 32 bit system.

Packages to install

# apt-get install build-essential git-core help2man diffstat texi2html texinfo cvs gawk subversion gawk

Addendum for i386:
# apt-get install python-psyco

Addendum for amd64:
# apt-get install ia32-libs

As the prompt will suggest, this command has to be executed as root; here's a description of the needed packages:

  • build-essential;
  • git-core;
  • help2man;
  • diffstat;
  • texi2html;
  • texinfo;
  • cvs;
  • gawk;
  • python-psyco;
  • subversion;
  • gawk;
  • ia32-libs.

The package manager will suggest to install extra needed packages: just confirm with Y and wait for the job to complete.

Creating the working directory

On your hard disk, choose a convenient place to store all your working environment; a directory with a short name like oe on your root directory may be convenient: 

$ cd ~
$ mkdir oe
$ cd oe

Installing bitbake

We install bitbake (BB for short) under the directory bitbake on your working directory; it's convenient to install BB from it's git repository to avoid outdated releases; at the time writing (April 22 2011) the stable release is 1.12.0.

Notes about git and openembedded

The command as above download the development branch of OE; if you want more control over the downloading process, consider as follows:

Checking the remote available branches

Once downloaded the development branch, you can issue the following command: 

~/oe$ cd openembedded
~/oe$ git branch -r
  origin/2011.03-maintenance
  origin/HEAD -> origin/master
  ...cut...
  origin/master
  ...cut...
  origin/stable/2009
  ...cut...
  origin/testing
  origin/testing-next
  origin/testing-next-test
  ...cut...

NOTE: Some cut has been performed to underline just the more interesting results. This will list all remotely available branches.

Using a particular branch

~/oe$ cd openembedded/
~/oe/openembedded$ git checkout origin/stable/2009 -b stable/2009
Branch stable/2009 set up to track remote branch stable/2009 from origin.
Switched to a new branch 'stable/2009'

Updating openembedded

From the moment you cloned OE on your system, some updates may occurs; to update your OE tree: 

~/oe$ cd openembedded/
~/oe/openembedded$ git pull

Create the local.conf file for openembedded

Look for a supported machine

Before editing your local.conf file, you have to discover if your machine is supported or not; actually (April 22, 2011) more than 200 machines are supported; I'm developing for a FriendlyARM Mini2440 and actually this machine is not present so I have to make some extra efforts to build for it.

Edit the file

A starting point for the local.conf file is already available; just made a copy into your build directory: 

fpiraneo@debian:~/oe/openembedded$ cd ~/oe
fpiraneo@debian:~/oe$ mkdir -p build/conf
fpiraneo@debian:~/oe$ cp openembedded/conf/local.conf.sample build/conf/local.conf

Edit the target file in ~/oe/build/conf/ to:

  • Define the source directory - Where BB will store all downloaded source; 
# Use this to specify where BitBake should place the downloaded sources into
DL_DIR = "${HOME}/sources"

A good directory may be: ${HOME}/oe/build/sources; so: 

DL_DIR = "${HOME}/oe/build/sources"
  • Define the recipe directory: 
# Delete the line below. Then specify which .bb files to consider for
# your build. Typically this will be something like BBFILES = "/path/to/openembedded/recipes/*/*.bb"
BBFILES := "${@bb.fatal('Edit your conf/local.conf: BBFILES')}"

As indicated the string between the quotes and replace with the directory containing the recipes, ie: 

BBFILES := "${HOME}/oe/openembedded/recipes/*/*.bb"
  • Here you can specify for which machine you're building for; enter the machine name you discovered on the precedent step; this will take care of all building options; 
# Uncomment this to specify a machine to build for. See the conf directory
# for machines currently known to OpenEmbedded. This will automatically take care
# of TARGET_ARCH
MACHINE = "mini2440"
  • The generation of binary locales takes lot of time, so I prefer to compile just the needed locales: 
# If ENABLE_BINARY_LOCALE_GENERATION is set to "1", you can limit locales
# generated to the list provided by GLIBC_GENERATE_LOCALES. This is huge
# time-savior for developmental builds. Format: list of locale.encoding pairs
# with spaces as separators.
GLIBC_GENERATE_LOCALES = "en_GB.UTF-8"
  • Again to save disk space and time, the generation of just a .tar image is suggested; a file system image can be generated later; 
# Add the required image file system types below. Valid are
# jffs2, tar(.gz|bz2), cpio(.gz), cramfs, ext2(.gz), ext3(.gz)
# squashfs, squashfs-lzma
IMAGE_FSTYPES = "tar"
  • On multiple core machine enable the parallel make may be a time saving feature; you can disable it in case of problems: 
# Uncomment this to enable parallel make.
# This allows make to spawn mutliple processes to take advantage of multiple
# processors. Useful on SMP machines. This may break some packages - we're
# in the process of marking these so let us know if you find any.
PARALLEL_MAKE = "-j 4"
  • remove the last two lines; this shows that you edited your local.conf file. 
# EDIT THIS FILE and then remove the line below before using! [REMOVE]
REMOVE_THIS_LINE:="${@bb.fatal('Read the comments in your conf/local.conf')}" [REMOVE]
